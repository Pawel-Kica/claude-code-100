From Google X: if the plan is a monkey reciting Shakespeare on a pedestal, train the monkey first. The pedestal is always buildable, so building it proves nothing. The monkey might be impossible, so it goes first.

Risk decides order, not difficulty. Hard-but-certain work can wait; maybe-impossible work cannot.

Given a project, feature, or plan:

1. List the parts. Label each one: pedestal (certain to work, only costs time) or monkey candidate (might be impossible, or possible but worthless).
2. For every candidate ask: if this fails, is the whole thing dead? The monkey is where the answer is yes.
3. Check for hidden monkeys outside the code: "will anyone pay", "will anyone use it", "does the data even exist".
4. Output: the monkey in one sentence, plus the cheapest ugly spike that tests it now. Everything else waits.

Several monkeys tie: pick the cheapest to test.
User is polishing pedestals: say it to their face.
